# Organization seed — one-time import

`org_seed.sql` is a **one-time** data import for the cleaned Australian
healthcare-facilities dataset. It is **not** a migration — running it on every
deploy is wasted work and may shadow operator-curated changes.

## When to run it

- Once per environment (dev, staging, prod), after the contact-enrichment
  schema migrations have applied (i.e. `parent_organization_id`,
  `organization_domains` exist).
- After clearing or re-baselining the `organizations` and `contacts` tables —
  e.g. you wiped contacts to re-import them through the new
  `upsert_contact_with_org_v2` pipeline.

## What it does

- Inserts 8 reference `organization_types` rows (idempotent — `ON CONFLICT DO
  UPDATE`).
- Inserts ~76 parent organisations (NSW Health and 17 LHDs, Queensland Health
  and 4 HHSs, state health departments, big private networks like Ramsay /
  St John of God / Calvary / etc.).
- Inserts ~2,366 facility organisations linked to their parent where
  derivable (1,097 / 45% have a known parent; the rest stay standalone).
- Inserts ~2,442 rows in `organization_domains` (one primary domain per org +
  parent aliases) so the `_resolve_org_by_domain` RPC helper can match
  inbound emails.

All inserts use `ON CONFLICT DO UPDATE` so re-running is idempotent.

## How to run it

### Option A — via Supabase SQL Editor (preferred for prod)

1. Open the project in the Supabase dashboard.
2. SQL Editor → paste the contents of `org_seed.sql` → run.

### Option B — via psql (preferred for dev / staging)

```bash
psql "$SUPABASE_DB_URL" -f supabase/seed/org_seed.sql
```

Where `SUPABASE_DB_URL` is the direct connection string for the target
project (Settings → Database → Connection string → URI).

### Option C — via Supabase CLI against a linked local

```bash
supabase db reset --no-seed     # if a clean slate is wanted
psql "$(supabase status -o env | grep DB_URL | cut -d= -f2)" -f supabase/seed/org_seed.sql
```

## Pre-clean for partial-state DBs

`organizations.domain` has a UNIQUE constraint. If the target environment
already has rows with domains the seed wants to insert (e.g. auto-created
by the pre-RPC intake code), the seed's pre-flight check aborts with a
clear `RAISE EXCEPTION` listing the conflicting domains.

Two recovery options — pick whichever fits your migration plan:

### Option A — adopt existing rows into the seed (lossy: id changes)

Updates the existing rows' ids to match the seed-expected ids, then runs
the seed normally. Foreign-key cascades from `contacts`,
`organization_domains`, etc. get updated automatically because their FKs
use `ON UPDATE CASCADE` — verify before running on prod.

```sql
-- Replace 'health.nsw.gov.au' / 'c6317794-…' with the actual domain + seed
-- UUID reported by the pre-flight error message, one block per domain.
BEGIN;
UPDATE public.organizations
   SET id = 'c6317794-387e-5d5d-a545-2ce10fe793b3'::uuid
 WHERE domain = 'health.nsw.gov.au';
COMMIT;
```

### Option B — delete auto-created intake placeholders, then run seed

Drops the offending rows entirely. Their contacts (and any FK-linked
records) move to a fresh row created by the seed. Use this if the
existing rows are clearly low-value placeholders (custom_fields tagged
`auto_created_from_intake`).

```sql
BEGIN;
DELETE FROM public.organizations
 WHERE custom_fields @> '{"auto_created_from_intake": true}'::jsonb
   AND domain IN (
     -- list of conflicting domains from the pre-flight error
     'health.nsw.gov.au',
     'wslhd.health.nsw.gov.au'
   );
COMMIT;
```

`contacts.organization_id` has `ON DELETE SET NULL` (or restricts) — verify
the FK behaviour before running the DELETE. If it RESTRICTs, re-link
contacts manually first or use Option A.

After either option, re-run `org_seed.sql`. The pre-flight passes; the
inserts proceed with the deterministic seed UUIDs.

## Regenerating

The file is generated by `scripts/build_org_seed.py` from
`db-backups/2026-02-09/main_data.sql`. **Do not edit by hand.** If the parent
map needs updates, edit the `PARENTS` dict in the script and re-run:

```bash
python3 scripts/build_org_seed.py
```

The output is committed for review and audit. Tests for the cleaner live in
`scripts/test_build_org_seed.py` — run with `python3 scripts/test_build_org_seed.py`.
